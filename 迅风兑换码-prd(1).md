# 迅风兑换码权限分离 & BPM 流程统一 需求文档

> 版本 v1.11 · 2026-03-06 · 草稿

---

## 产品定位

**是什么**：在迅风系统中，对兑换码的生成行为进行权限管控，所有新增兑换码均须经过 BPM 审批方可生成码值，同时对各角色的查询和下载权限进行精细化分离，从源头防止兑换码私下售卖。

**面向谁**：
- **运营人员**：创建/修改兑换码活动、发起 BPM 生成审批、管理道具定价
- **客服人员**：查询单个兑换码或用户的激活状态，不可批量下载未激活码值
- **BPM 审批人**：在 BPM 系统中审批兑换码生成请求（对接飞书审批流）

**不做什么**：
- 不改变兑换码的兑换侧逻辑（用户端兑换体验不变）
- 不提供自助申诉/重新审批通道（被拒绝后需联系系统管理员）
- 不包含数据大屏/BI 报表前端开发（数据同步至大数据平台，由大数据侧建模）


---

## 用户故事

- 作为**运营人员**，我希望创建兑换码活动并申请生成码值，以便开展营销/补偿等活动
- 作为**运营人员**，我希望只能下载自己生成批次的未激活码值，以便防止同事间私自传播未使用码
- 作为**运营人员**，我希望可以查看所有人已激活的码值记录，以便追溯激活情况
- 作为**客服人员**，我希望通过单个码值或用户 ID 快速查到激活状态，以便处理玩家兑换客诉
- 作为**审批人**，我希望在 BPM/飞书中看到完整的活动信息和预算价值，以便评估是否批准本次生成请求

## 兑换码类型说明

系统支持两种兑换码类型，两者在生成方式、使用逻辑和审批流程上存在本质差异：

### 单人兑换码（一码一人）

系统为每个兑换码活动批量生成**唯一码值**，每个码只能被一个角色激活一次，激活后码值失效。

- **生成方式**：运营先创建活动，再单独发起生成申请（走 BPM 审批），审批通过后系统生成对应数量的随机码值
- **兑换方式**：玩家输入具体码值（如 `XFJM2A8K`）进行兑换
- **使用限制**：每个码值只能被激活一次；同一角色在同一活动内可配置兑换次数上限
- **典型场景**：定向发码（补偿、KOL 合作、营销拉新）、需要控制领取人群的活动

### 复用兑换码（一码多人）

运营人员自定义一个**兑换口令**，所有知道口令的玩家均可兑换，直到达到兑换次数上限为止。

- **生成方式**：创建活动即触发 BPM 审批，审批通过后口令生效；无需单独生成步骤
- **兑换方式**：玩家输入口令（如 `SPRING2026`）进行兑换
- **使用限制**：整个口令有可兑换次数上限（可设为不限）；达到上限后所有人均无法再兑换
- **典型场景**：直播间口令码、社群推广码、节日全服礼包

### 两种类型核心差异对比

| 对比维度 | 单人兑换码 | 复用兑换码 |
|---------|-----------|-----------|
| 码值形态 | 系统生成的随机唯一码（如 `XFJM2A8K`） | 运营自定义口令（如 `SPRING2026`） |
| 激活人数 | 每码仅限 1 人激活 | 同一口令多人可用，受次数上限控制 |
| 审批触发时机 | 创建活动后，**单独发起**生成申请时触发审批 | **创建活动时**即触发审批（无独立生成步骤） |
| 审批拒绝后 | 活动**保留**，本次生成不成功；不可重新申请（联系管理员） |同单人兑换码
| 审批中是否可修改 | 除礼包配置和用途外，其他字段**可修改** |除礼包/价值/次数/用途外其他字段**可修改** |
| 单码查询方式| 可通过角色 ID、角色名称、码值查询 | 仅支持通过口令精确查询 |

## 功能模块

### 模块一：道具管理【运营角色】

运营人员为每款游戏维护道具单价表，单价用于自动计算兑换码的「单码价值」，作为 BPM 审批时的预算依据。

**核心交互**

| 操作 | 触发方式 | 响应行为 |
|------|---------|---------|
| 新增道具 | 点击「新增」按钮 | 展开行内表单或打开侧边弹窗，填写字段后保存 |
| 批量上传 | 点击「批量上传」，选择 Excel 文件 | 上传后展示解析预览，确认后写入；格式错误逐行标红提示 |
| 批量删除 | 勾选多行后点击「批量删除」 | 弹出确认弹窗（"确认删除 N 条道具？"），确认后删除 |
| 修改单条 | 点击操作列「修改」 | 弹窗或行内编辑，保存后刷新当前行 |
| 删除单条 | 点击操作列「删除」 | 确认弹窗后删除，关联兑换码活动中已引用的道具不受影响（价格快照） |

**列表字段**

| 字段 | 说明 |
|------|------|
| 类型 ID | 道具类型标识 |
| 类型名称 | 道具类型名称 |
| 物品 ID | 唯一标识 |
| 物品名称 | 显示名称 |
| 最大数量 | 单次礼包可配置的上限数量 |
| 价格（元） | 道具单价，手动维护 |
| 操作 | 修改 / 删除 |

**页面状态**

- **加载中**：表格骨架屏
- **空状态**：「暂无道具数据，请点击新增或批量上传」
- **加载失败**：「数据加载失败，请刷新重试」+ 刷新按钮

**边界情况**

- 价格字段仅允许正数，最多两位小数；非法输入实时提示「请输入有效金额」
- 批量上传文件大小上限：待确认
- 道具被兑换码活动引用后，删除道具不影响历史活动的单码价值计算（价格快照机制）

- 查询无结果：「未找到匹配的兑换码记录」

### 模块三：生成兑换码【运营角色，需 BPM 审批】

所有兑换码的码值生成均须提交 BPM 审批。审批对接飞书，审批结果通过飞书消息通知申请人。

#### 3.1 单人兑换码（一码一人）

**活动列表字段**

| 字段 | 说明 |
|------|------|
| 活动 ID | 活动唯一标识 |
| 活动标题 | 活动名称 |
| 所属游戏 | 游戏名称 |
| 用途 | 活动用途标签 |
| 单码价值 | 约 ¥XX 元（根据礼包自动计算） |
| 已生成 / 审批中数量 | 历史已生成总数 / 当前审批中数量（如 `5,000 / 0`） |
| 有效期 | 开始时间–结束时间 |
| **活动状态** | 进行中 / 待开始 / 已结束 |
| 创建人 | 创建活动的运营人员 |
| 操作 | 查看 / 查看码值 / 生成 / 修改 / 复制（依活动状态控制按钮可用性） |

##### 3.1.1 新增兑换码活动

运营人员在迅风中创建活动配置，创建后活动处于「待生成」状态，尚未产生码值。

**活动配置字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| 所属游戏 | 下拉选择 | 必填 |
| 区服 | 下拉选择 | 必填 |
| 渠道 | 下拉选择 | 必填 |
| 版本 | 输入 / 下拉 | 必填 |
| 开始时间 | 日期时间选择器 | 必填 |
| 结束时间 | 日期时间选择器 | 必填，须晚于开始时间 |
| 兑换上限 | 数字输入 | 每人在本活动最多兑换 N 次 |
| 循环周期 | 数字输入 | 每 N 天可兑换 1 次；0 或空表示不限 |
| 单向排斥活动 | 可搜索 + 勾选多选 | 提供搜索框（按活动 ID 或名称筛选）及活动列表勾选；用户勾选多个活动作为排斥项；若用户已兑换指定活动，则不能参与本活动 |
| 活动标题 | 文本输入 | 必填 |
| 礼包配置 | 道具选择器 + 数量 | 从道具管理表中选择，支持多条,每次点击添加，自动增加一条 |
| 单码价值 | 自动计算（只读） | 根据礼包道具单价 × 数量汇总；显示「约 ¥XX 元」 |
| 用途 | 单选 + 二级信息 | 见下方用途枚举 |
| 备注 | 文本域 | 选填 |

**用途枚举及二级字段**

用途分为三个一级类别，如下。

| 一级类别 | 二级选项 / 说明 |
|----------|-----------------|
| **免费活动** | **营销活动**：拉新活动、广告投放、日常促活、节日礼包等；**售后补偿**：客诉处理、系统故障补偿；**内部测试**：开发测试、员工福利；**商务合作**：外部渠道合作、大客户赠送 |
| **创作者激励** | 填写激励活动名称（非必填） |
| **售卖** | fanbook 商城；微信商城；**抖音商城**（选择官方店/合作店）；**商务合作**（填写渠道名称）；**KOL 合作**（填写 KOL 名称） |

- **免费活动**：运营在「用途」下先选一级「免费活动」，再选二级：营销活动、售后补偿、内部测试、商务合作之一；含义以括号内说明为准。
- **创作者激励**：选一级「创作者激励」后，可填写激励活动名称，为非必填。
- **售卖**：选一级「售卖」后，再选二级：fanbook 商城、微信商城、抖音商城（官方店/合作店）、商务合作、KOL 合作之一；抖音商城需区分官方店/合作店，商务合作与 KOL 合作需填写渠道名称或 KOL 名称。

##### 3.1.2 修改兑换码活动

**修改权限规则**

| 生成状态 | 可修改字段 |
|---------|---------|
| 待生成（未发起过审批） | 所有字段 |
| 审批中 | 除「礼包配置」和「用途」外，其他字段均可修改 |
| 已生成（已生成过码值） | 除「礼包配置」和「用途」外，其他字段均可修改 |
| 已拒绝 | 不可修改任何字段（需联系管理员处理） |

> 修改权限不限本人，任何运营人员均可修改（防止休假等情况下无人处理）

> **活动状态 vs 生成状态 vs 审批状态**：「活动状态」是**活动级别**的时间维度状态（未开始 / 进行中 / 已结束），由活动的开始时间与结束时间决定；「生成状态」是**活动级别**的码值生成维度状态（待生成 / 审批中 / 已生成 / 已拒绝），由最新一次生成记录的结果推导而来；「审批状态」仅用于**生成记录**粒度，表示某次具体 BPM 审批的结论（见 3.1.4）。一个活动可多次发起生成申请，每次生成独立形成一条记录。

##### 3.1.3 生成兑换码（发起 BPM 审批）

单人兑换码支持两种发起方式：**单活动生成**（在某一活动的操作列点击「生成」，仅该活动进入一条 BPM 审批）与**批量生成**（在活动列表勾选多个活动后点击「批量生成审批」，所有勾选活动合并为**一条**飞书审批）。以下「生成表单字段」与「BPM 审批单呈现信息」针对单活动生成；批量生成时的审批单结构见本小节末尾及模块四 4.1。

**生成表单字段**

| 字段 | 说明 |
|------|------|
| 本次生成数量 | 手动填写，必填 |
| 本次预计生成价值（商业化） | 自动计算：本次数量 × 单码价值，只读 |
| 总计生成价值（商业化） | 自动计算：历史已生成价值 + 本次预计生成价值，只读 |

**BPM 审批单呈现信息（提交到飞书）**

*活动信息部分*
- 所属游戏、区服、渠道、版本、开始/结束时间、每人兑换上限、循环周期、单向排斥活动、活动标题、礼包配置（道具名称 + 数量）、单码价值、用途

*本次生成信息*
- 生成时间
- 本次生成个数
- 本次预计生成价值（元）
- 历史已生成价值（元）
- 总预计生成价值 = 历史已生成 + 本次预计（元）
**批量生成时的 BPM 审批单（一条飞书审批，含汇总）**

当运营通过「批量生成审批」勾选多个活动并提交时，系统生成**一条** BPM/飞书审批单，审批单内容包含：

- **汇总信息**：申请时间、申请人、关联活动数量、本单总生成数量（各活动本次生成数量之和）、本单总预计生成价值（商业化）（各活动本次预计价值之和）。
- **各活动明细**：以表格或列表形式逐条展示本单内每个活动的活动信息（活动标题、单码价值等）及本次生成信息（本次生成数量、本次预计生成价值等），便于审批人逐项核对。
- 审批通过后，系统按明细为每个活动分别生成对应数量的码值；审批拒绝则本单内所有活动均不生成码值。

**审批结果处理**

| 审批结果 | 系统行为 |
|---------|---------|
| 通过 | 生成对应数量的码值；生成记录状态变更为「已同意」；飞书通知申请人 |
| 拒绝 | 本次生成记录标记为「已拒绝」；不生成任何码值；活动保留；**不可重新发起申请**（需联系管理员）；飞书通知申请人 |
| 审批中 | 码值保持「待审核」状态，活动正常可见但码值未产生 |

##### 3.1.4 生成记录与日志

**生成记录字段**：生成时间、生成人、本次生成数量、本次生成价值、审批状态、审批链接

**查看码值入口**：活动列表操作列「查看码值」按钮，点击后打开当前活动的码值弹窗（查看=可下载，无需跳转独立页面）。

**批量上传禁用 / 批量上传解禁（功能说明）**

- **入口与范围**：两个按钮位于**单人兑换码 · 活动列表**页、列表上方（与筛选栏同一行，右侧）。操作对**全部所有单人兑换码活动**的码值生效，不区分活动、不区分批次；即上传文件中的码值会与系统中所有单人兑换码活动的码值做匹配，匹配到的即执行禁用或解禁。
- **批量上传禁用**
  - 运营或客服在活动列表页点击「批量上传禁用」，选择本地的 Excel 文件（.xlsx / .xls）。
  - 文件需包含「兑换码」列（或首列为码值），每行一个码值；系统解析后与**全部单人兑换码活动下所有批次**的码值做匹配。
  - 对文件中**且存在于系统中（任意活动、任意批次）**的码值批量标记为「已禁用」，即这些码值立即失效、不可再兑换；文件中不存在的码值忽略（可选：在结果中提示「未匹配 N 条」）。
  - 上传后建议：先展示解析预览（如总行数、有效码数量、匹配数量），确认后执行；若存在格式错误可对异常行标红提示。执行完成后提示「已禁用 N 条」或「成功 N 条，未匹配 M 条」。
- **批量上传解禁**
  - 运营或客服在活动列表页点击「批量上传解禁」，同样上传含「兑换码」列的 Excel 文件。
  - 对文件中**且存在于系统中且当前为已禁用**的码值批量解除禁用，状态恢复为「未禁用」，可再次兑换；已为未禁用的码值无需处理；不存在的码值忽略。
  - 上传后同样建议解析预览与确认，执行完成后提示「已解禁 N 条」或「成功 N 条，未匹配/已为未禁用 M 条」。
- **权限**：运营与客服角色均可见并可操作。
- **与单条操作关系**：单条「禁用/解禁」在「单码查询」或「查看码值」弹窗内对单条码值操作；批量上传在活动列表页对全部活动生效，二者效果一致（均为码值禁用状态变更），仅操作粒度不同。

### 模块五：兑换码报表【大数据平台，仅规范数据口径】

> 报表前端由大数据平台开发，本文档仅定义数据维度和指标口径。

**业务维度**

| 维度 | 说明 |
|------|------|
| 游戏 | 所属游戏 |
| 用途场景 | 对应活动用途（营销拉新/售后补偿等） |
| 免费/付费 | 免费活动 vs 售卖渠道 |
| 付费渠道 | 售卖时所选的渠道 |
| 活动创建人 | 创建该活动的运营人员 |
| 码值生成人 | 发起 BPM 生成审批的操作人 |

**时间维度**：月度、年度

**核心指标**

| 指标 | 定义 |
|------|------|
| 生成金额 | 审批通过并生成的码值总价值（数量 × 单码价值） |
| 实际兑换金额 | 已被用户激活兑换的码值总价值 |
| 实际兑换人数 | 已完成兑换（激活）的兑换次数，不去重 |

**考核数据通知**

上述指标与维度的考核数据，需按**日、周、月**的形式，通过**飞书群通知**的方式告知相关成员。

**日报内容（梳理）**

日报为按**日**维度的考核与变更汇总，通过飞书群推送。建议包含以下部分：

| 部分 | 内容说明 |
|------|----------|
| **一、当日考核数据** | 按模块五口径，统计**当日**（自然日或统计日）的指标与维度数据。指标：当日**生成金额**、当日**实际兑换金额**、当日**实际兑换人数**。维度：可按游戏、用途场景、免费/付费、付费渠道、活动创建人、码值生成人等拆解展示（具体拆解粒度由实现与产品约定）。 |
| **二、当日后台更改记录** | 若当天发生过对「审批中」或「已审批完成」数据的**后台手动更改**（即 PRD 规定不可在线上修改的锁定字段等），系统在更改完成后将**当天的更改内容**同步到日报。建议包含：更改时间、变更对象（活动 ID/申请单号等）、变更类型或字段、变更前后值（若可脱敏展示）、操作人/申请人等，便于考核与审计追溯。无当日后台更改时，本部分可为空或标注「无」。 |

周报、月报可在此基础上按周、月聚合指标与维度，并可选汇总周期内的后台更改条数或摘要。

---

### 审批中 / 已审批完成数据的修改与日报同步

对于已进入**审批中**或**已审批完成**的迅风兑换码（活动或生成记录），**原则上不支持修改**，与 PRD 中各模块「可修改字段」规则一致。

在**紧急情况**下，可联络技术人员进行**后台更改**。此处「更改」指：原 PRD 文档中规定的、**不可在线上系统中修改的部分**（例如单人兑换码的礼包配置、用途；复用兑换码的口令、礼包配置、单码价值、可兑换次数、用途等锁定字段；或已审批完成后的其他仅允许后台变更的数据）。

若进行了上述手动后台更改，系统应在更改完成后，将**当天的更改内容**同步到**日报**中，便于考核与审计追溯。

## MVP 范围

**核心路径 v1 必须有**
- 渠道、区服可以替代
- 复用兑换码：新增活动（即触发审批）→ 审批通过 → 活动生效
- 客服查询：单个码值 / 用户激活状态查询
- 道具管理：新增 / 批量上传 / 修改 / 删除道具单价

**增强功能（后续迭代）**
- 兑换码报表（大数据平台接入）
- 标价优化：按月度/每次活动投放总价值申报，减少申请人的填写负担
